# DECISIONS — 의사결정 로그

> 헌법 변경/기술 선택은 모두 여기에 새 항목으로 추가. 기존 항목은 수정하지 않고 후속 항목으로 추가.
> 채워진 sample: 정답 PR 시리즈 머지 후 운영진이 별도 공유.

## 형식

```markdown
### D-NNN. (한 줄 결정)

- **일자**: YYYY-MM-DD
- **상태**: 채택 / 보류 / 폐기
- **배경**: 무엇이 문제였나? (Problem)
- **선택지**: 본 결정에서 고려한 옵션 2~3개 + 각각의 trade-off (Options)
- **결정**: 최종 선택 + 이유 (Decision)
- **영향**: 어떤 코드/문서/프로세스에 영향 가는가
- **재평가**: 언제 / 어떤 신호 보면 다시 검토하는가
```

## 의사결정

### D-001. (예: 추천 프로젝트 1번 — 운영 티켓 시스템 선정)

- **일자**: 2026-MM-DD
- **상태**: 채택
- **배경**: Week 9 5일 안에 6 공통 필수 기능 + 라이프사이클 5 단계를 모두 시연해야 함. 도메인이 너무 익숙하면 학습 효과 적고, 너무 낯설면 5일 안에 완주 어려움.
- **선택지**:
  1. 운영 티켓 시스템 — 도메인 단순 + 6 공통 필수 기능 매핑 자연스러움
  2. 커머스 주문 후처리 — 트랜잭션 풍부하지만 결제 연동이 시간 잡아먹음
  3. 자율 주제 — 4명 합의 비용 ↑
- **결정**: 1번. 5일 일정에 가장 적합 + 운영자/신청자 페르소나 구분이 권한 기능에 자연스럽게 매핑.
- **영향**: PRD.md, CLAUDE.md 도메인 컨텍스트, src/ 패키지 구조
- **재평가**: Phase 3 진입 시 도메인 확장 (Comment + AI 답변 초안) 필요 시

### D-002. (예: 라이프사이클 5 단계 책임자 분담)

- **일자**: 2026-MM-DD
- **상태**: 채택
- **배경**: 5단계를 1명이 다 못함. 4명에 5단계라 1명이 2단계 겸함.
- **선택지**: 생략 — 강제 분담 정책.
- **결정**: 기획 (팀원 A) / 코딩 (팀 전원) / 테스트 (팀원 B) / 리뷰 (팀원 C) / 운영 (팀원 D). 코딩만 팀 전원 공동.
- **영향**: LIFECYCLE-COVERAGE.md 매트릭스
- **재평가**: 누가 명백히 막힘 발생 시 swap

### D-003. (예: 기술 스택 — Spring Boot 3.3 + Java 21 + JPA + Redis)

- **일자**: 2026-MM-DD
- **상태**: 채택
- **배경**: 부트캠프 학습 자산이 Spring Boot 3.3 / Java 21 기준. 다른 스택은 학습 비용 ↑.
- **선택지**: Kotlin / Spring Webflux / Quarkus 등 검토 후 모두 기각.
- **결정**: 학습 자산 그대로 활용. 라이브러리 선택은 CLAUDE.md 코딩 규칙 따름.
- **영향**: build.gradle, application.yml, 패키지 구조
- **재평가**: Phase 3 진입 시 Reactive / Native image 등 검토 가능
